# 产品与架构说明

## 1. 用户定位

- 宠主：需要短时上门喂养、遛狗、溜猫、临时寄养的人。
- 兼职铲屎官：希望利用碎片时间赚零花钱、提供宠物照护服务的人。

## 2. MVP 核心闭环

### 宠主侧

1. 登录平台并切换为宠主模式。
2. 在地图上选中服务地点，填写服务类型、时间、价格和宠物说明。
3. 提交订单，进入待接单状态。
4. 等待附近铲屎官接单，后续进入聊天、服务、评价流程。

### 铲屎官侧

1. 登录并切换为铲屎官模式。
2. 浏览附近 5km 的待接任务。
3. 选择合适订单并立即接单。
4. 后续完成服务并沉淀评价与服务记录。

## 3. 数据设计

### 核心实体

- `users`：用户资料、角色、手机号、最后坐标、评分、完成单数。
- `pets`：宠物资料，可一对多归属到用户。
- `orders`：订单中心，记录状态流转、位置、价格、时长等。
- `posts`：社区动态内容、媒体资源、标签、互动数据。
- `reviews`：订单完成后的评价体系，积累信任。

### 地理查询建议

- 用户发单时，前端拿到高德地图点击后的 `(lng, lat)`。
- 后端写入 PostgreSQL/PostGIS 的 `GEOGRAPHY(POINT, 4326)` 字段。
- 查询附近任务时使用 `ST_DWithin(location, ST_MakePoint(lng, lat)::geography, 5000)`。

## 4. 当前代码策略

为保证从空目录能快速看到可用原型，当前后端 API 采用“真实接口结构 + 内存演示数据”的方式：

- 前端已经按真实接口契约组织请求。
- 后端路由已具备正式化命名和请求模型。
- 数据库初始化 SQL 已按 PostGIS 方案给出。
- 后续切换真实数据库时，只需要替换仓储层即可，不必重写页面和接口协议。

## 5. 下一阶段优先级

1. 登录接入微信开放平台 + 阿里云/腾讯云短信。
2. 订单状态机：待接单、已接单、服务中、已完成、取消。
3. WebSocket 即时通知 + 聊天。
4. OSS 上传与媒体审核。
5. 支付、保证金、评价和风控。
